## Overview

This project fine-tunes **TrOCR-Large (Stage 1)** for optical character recognition on cropped table-cell images. Training was performed on AWS SageMaker using distributed training across 4 GPUs, achieving near-perfect OCR accuracy on the target domain.

## 1. Dataset

### Source & Labeling

Images were generated by an internal pipeline that produces cropped table-cell screenshots. OCR ground-truth labels were generated automatically using **AWS Bedrock** with the `qwen.qwen3-vl-235b-a22b` model, with a domain-specific system prompt that enforces exact transcription.

### Data Pipeline

```
Internal Pipeline → Cell Cropping → MD5 Deduplication → Bedrock OCR Labeling → Train/Val Split
```

### Dataset Statistics

| Split      | Count       | Percentage |
|------------|-------------|------------|
| Training   | 103,232     | 80%        |
| Validation | 25,809      | 20%        |
| **Total**  | **129,041** | **100%**   |

- **Image dimensions**: avg. 36 px height × 169 px width (small cell crops)
- **Format**: PNG/JPG, converted to RGB
- **Deduplication**: MD5 hash-based; duplicates mapped back to canonical labels before splitting
- **Label examples**: `"Apr27-Oct27"`, `"1,234.56"`, `"Header Text"`

## 2. Model Architecture

**Base Model**: `trocr-large-stage1`

The model is a Vision Encoder-Decoder (`VisionEncoderDecoderModel`) combining a ViT image encoder with a TrOCR autoregressive text decoder.

### Encoder: Vision Transformer (ViT-Large)

| Parameter        | Value     |
|------------------|-----------|
| Input resolution | 384 × 384 |
| Patch size       | 16 × 16   |
| Hidden size      | 1024      |
| Attention heads  | 16        |
| Hidden layers    | 24        |
| Activation       | GELU      |

### Decoder: TrOCR Transformer

| Parameter               | Value  |
|-------------------------|--------|
| Model dimension         | 1024   |
| Attention heads         | 16     |
| FFN dimension           | 4096   |
| Decoder layers          | 12     |
| Dropout                 | 0.1    |
| Vocabulary size         | 50,265 |
| Max position embeddings | 1024   |

### Image Preprocessing

- Resize to 384 × 384
- Rescale by factor `1/255`
- Normalize: mean `[0.5, 0.5, 0.5]`, std `[0.5, 0.5, 0.5]`

## 4. Loss Function & Optimization

- **Loss**: Cross-entropy (causal language modeling) on decoder outputs
- **Label masking**: PAD tokens replaced with `-100` and excluded from loss computation
- **Gradient clipping**: Max norm 1.0 to prevent exploding gradients
- **AMP**: Enabled for forward/backward passes with `GradScaler` for numerical stability

### Notable Engineering Fix

`TrOCRSinusoidalPositionalEmbedding` stores its weights as a plain Python attribute rather than a registered PyTorch buffer. As a result, calling `.to(device)` on the model silently skips these weights, causing a device mismatch at runtime. This was patched explicitly in `sm_train/train.py`:

```python
embed_pos = model.decoder.model.decoder.embed_positions
embed_pos.weights = embed_pos.get_embedding(
    embed_pos.weights.shape[0],
    embed_pos.embedding_dim,
    embed_pos.padding_idx,
).to(device)
```

The ViT pooler was also frozen (it is unused in the seq2seq architecture) to eliminate DDP warnings about unused parameters.

## 5. Checkpointing & Evaluation

- **Intra-epoch checkpoints**: Every 200 steps (saved on rank 0 only)
- **End-of-epoch checkpoints**: After each epoch completes
- **Evaluation frequency**: Every 2 epochs on the full validation set
- **Best model selection**: Saved automatically whenever a new minimum CER is achieved
- **Checkpoint contents**: Model weights, optimizer state, LR scheduler state, AMP scaler state, best CER value

### Validation Metric: Character Error Rate (CER)

```
CER = (Substitutions + Deletions + Insertions) / Total reference characters
```

Predictions are generated using greedy decoding (`num_beams=1`, `max_new_tokens=64`) with special tokens stripped from output.

## 7. Inference

### Generation Configuration

| Parameter           | Value              |
|---------------------|--------------------|
| Max new tokens      | 64 (128 in handler)|
| Num beams           | 1 (greedy)         |
| No-repeat n-gram    | 3                  |
| Decoder start token | `<CLS>` (ID: 2)   |

### SageMaker Inference Handler (`trained_models/V1/best/code/inference.py`)

- **Input**: JSON with base64-encoded images — `{"images": [{"image_data": "...", "hash": "..."}]}`
- **Batch processing**: 32 images per chunk, with GPU cache cleared between chunks
- **Output**: `{hash: ocr_text, ...}` plus metadata (successful/failed counts, GPU memory usage)
- **Optimization**: `torch.compile()` for PyTorch 2.0+ inference acceleration
- **Error handling**: Graceful GPU OOM fallback, failed image filtering, health check endpoint

### Observed Limitation

Source images average ~36 px in height. When resized to 384×384 for the ViT processor, fine detail can be lost. Testing in `test_inference.ipynb` showed that applying a **4× upscale** before processing improves recognition quality on very small crops.

## 8. Deployment

```
├── preprocessor_config.json
├── tokenizer.json
├── vocab.json
├── merges.txt
├── tokenizer_config.json
├── special_tokens_map.json
└── code/
    ├── inference.py
    └── requirements.txt
```

**Deployment steps**:

1. Upload archive to S3: `aws s3 cp model.tar.gz s3://<bucket>/<prefix>/`
2. Create SageMaker model endpoint

## Summary

This is a complete, end-to-end OCR fine-tuning pipeline built for domain-specific table cell data:

- **Data**: 129K domain-specific cell-crop images labeled via Bedrock VLM OCR
- **Training**: Distributed across 4× A10G GPUs, TrOCR-Large base, 10 epochs
- **Result**: 0.000765 CER — near-perfect accuracy on the target domain
- **Deployment**: SageMaker-ready with batch inference handler included

The primary area for further improvement is handling very small source images (sub-40 px height), where pre-upscaling before the ViT processor has demonstrated meaningful benefit.
